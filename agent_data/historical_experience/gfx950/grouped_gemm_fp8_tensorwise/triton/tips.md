# Historical Experience Tips

Operator: grouped_gemm_fp8_tensorwise — Backend: TRITON — GPU: gfx950


### 2026-04-25 01:34 round-2 — ACCEPTED
- Context: gfx950 + Triton FP8 GEMMs (grouped or non-grouped) where `tl.dot` is invoked with FP8 operands and an FP32 accumulator.
- Signal: Removing `input_precision="ieee"` from the two `tl.dot` call-sites yielded a noise-bounded forward/backward improvement (bwd geomean -0.55%, fwd flat) with no IR-level regression; the kwarg forces a slow IEEE-compliant accumulation path that the AMDGCN backend would otherwise specialize.
- Takeaway: On gfx950 Triton FP8 `tl.dot`, omit `input_precision="ieee"` (let the backend pick its native MFMA accumulation) unless the kernel is on a numerically-strict reference path.
- Applicability: Apply to any gfx950 Triton FP8/BF16 GEMM with FP32 accumulator. Do NOT apply when the kernel is the bit-exact reference used to validate quantization correctness, or when the op explicitly requires IEEE-conformant rounding.

### 2026-04-25 01:34 round-3 — ACCEPTED
- Context: gfx950 + Triton persistent FP8 GEMM kernels with VGPR≈128, scratch≤16B, num_warps=8 (i.e. compute-bound MFMA-heavy kernels that fit comfortably in the 512 VGPR/EU register file).
- Signal: Setting `waves_per_eu=2` (vs the default 0/auto) on the forward persistent kernel produced +0.74% Forward TFLOPS and SGPR grew 80→112 with no VGPR or scratch increase — direct evidence the relaxed cap let the compiler hold more loop-state in registers. The same lever applied to a structurally similar wgrad kernel (R4) was a no-op, indicating sensitivity is to per-call latency headroom + clean register footprint, not just shape.
- Takeaway: On gfx950 Triton MFMA-heavy persistent kernels with VGPR≤128 and scratch≤16B, explicitly cap `waves_per_eu=2` to give the AMDGCN scheduler ~256 VGPR/wave to spend on loop-state; the auto setting under-allocates.
- Applicability: Apply when the kernel is compute-bound, fits in <=128 VGPR with clean scratch, and has per-call duration headroom. Do NOT apply when the kernel already spills (scratch >=64B) or VGPR is at the 256 ceiling — the cap then locks in fewer waves/EU without giving the compiler more registers to use.

### 2026-04-25 01:34 round-7 — ROLLED_BACK
- Context: gfx950 + ROCm HIP/Triton kernels that attempt to fuse multi-pass reductions (e.g. amax → scale → cast) using an intra-kernel global-atomic counter as an inter-block barrier so a "last block" can broadcast a reduction result to other blocks in the same launch.
- Signal: R7 (HIP fused amax+cast with atomic-counter barrier) and R18 (Triton fused add+amax with `tl.atomic_max` + barrier) both failed correctness checks (R7 core-shape regressed -50.55%, R18 Check≠PASS). ROCm gives no cross-block ordering guarantee for global atomics inside a single kernel launch — blocks released by the "last block" still see partial results.
- Takeaway: Do not implement inter-block barriers via in-kernel global-atomic counters on ROCm; reductions that must broadcast a result to all blocks need a kernel-boundary fence (split into two kernel launches), not an intra-kernel barrier.
- Applicability: Reuse this rule whenever fusing amax/scale/cast/normalize chains on AMD GPUs. Does not apply to (a) per-tile/per-program local atomics that DO NOT require cross-block ordering (e.g. `tl.atomic_add` accumulating disjoint output tiles), or (b) NVIDIA where cooperative-grid sync gives the missing guarantee.

### 2026-04-25 01:34 round-10 — ROLLED_BACK
- Context: gfx950 + FP8 tensorwise grouped GEMM where the project's tensorwise-scale layout (per-tensor scalar `scale_inv`, FP8 e4m3/e5m2 surfaces with project-specific stride conventions) is fed to a vendor backend (hipBLASLt R10, AITER/CK R37) via runtime gate.
- Signal: R10 hipBLASLt swap failed correctness immediately on the wgrad path; R37 AITER swap passed correctness but did not beat the in-house Triton kernel (Forward TFLOPS regressed vs current best). Vendor FP8 grouped-GEMM kernels assume their own scale-layout contract (per-channel/per-block scales, opposite operand transposition, different scale-inv semantics).
- Takeaway: Before flipping a runtime gate to a vendor FP8 grouped-GEMM backend (hipBLASLt, AITER/CK), preflight a one-shot scalar-correctness probe at module load against the in-house tensorwise reference and only enable the gate if the probe passes within FP8 tolerance.
- Applicability: Apply whenever introducing a vendor backend behind a project-managed FP8 quantization scheme. Does not apply to BF16/FP16 vendor dispatch (vendor scale contract is not in play) or to pure functional kernels with no scale-layout coupling.

### 2026-04-25 01:34 round-47 — ROLLED_BACK
- Context: gfx950 + Triton FP8 GEMM kernels where one call-site (e.g. dgrad) reads an operand with a strided/transposed pattern and the AMDGCN backend reports persistent register-spill scratch (e.g. dgrad call-site spilled 84B vs 16B on the forward call-site of the same shared kernel).
- Signal: Five rounds (R38 sibling-kernel split, R39 num_warps lift, R40 BLOCK_K halve, R45 joint autotune with separate keys, R47 per-call FP8 transpose into a companion buffer) all failed to eliminate the dgrad spill. R47 specifically paid ~0.27 ms/call HBM round-trip on the per-call transpose; the savings from reduced spill did not exceed the new HBM cost.
- Takeaway: Stride-induced per-iteration register-spill on a Triton GEMM consumer is best fixed at the producer (one-time storage-layout choice when the FP8 surface is materialized), not at the consumer via a per-call transpose, sibling-kernel split, or autotune.
- Applicability: Apply when a kernel's spill is correlated with a specific operand's stride pattern AND the producer of that operand is in your control. Does not apply when the operand layout is dictated by an upstream framework you cannot modify, or when the consumer's spill is independent of input stride (e.g. caused by accumulator size or num_warps).

### 2026-04-27 06:11 round-57 — ACCEPTED
- Context: gfx950 + Triton FP8 (or any quantized) op where the SAME bf16/fp16 input tensor is consumed by multiple kernel call-sites within one optimizer step and the producer is a per-call quantize launch (e.g. `quantize_fp8_tensorwise` / `unary_kernel<bf16->fp8>` + `reduce_row_kernel<AbsMaxOp>` + `compute_scale_from_amax_kernel`).
- Signal: Stacking persistent weakref-keyed quant caches on the FP8 weight (R56, +19.83% step geomean), activation (R57, +16.55% additional), and grad_out (R77) surfaces was the ONLY structural win in 100 rounds and produced cumulative +36.55% step geomean / +78% Forward TFLOPS. Cache key is `(data_ptr, shape, dtype, _version, device)` plus a `weakref` on the source tensor; `_version` only ticks on optimizer step / in-place mutation, so the same quant output is returned to all consumers within one training iteration. rocprofv3 confirmed `unary_kernel<bf16->fp8>` launches drop ~33-50%/step.
- Takeaway: When a producer kernel is fired multiple times per step with stable inputs, memoize its output in a process-singleton dict keyed on `(data_ptr, shape, dtype, _version, device)` + a weakref-liveness guard so cache invalidates exactly when the source tensor mutates or is freed.
- Applicability: Apply to producer ops whose output is a deterministic function of an input tensor that has stable identity within a step (weights between optimizer.step calls, activations between fwd-and-bwd, grad_out between dgrad-and-wgrad consumers). Do NOT apply to inputs that mutate without ticking `_version` (raw storage writes via `data_ptr`, custom C++ ops bypassing autograd), or to outputs whose downstream consumer mutates them in-place.

### 2026-04-27 06:12 round-81 — ROLLED_BACK
- Context: PyTorch autograd Functions whose backward returns a per-call tensor for a leaf-parameter input slot, where the leaf's framework-emitted `AccumulateGrad` add becomes a top-N profile consumer (e.g. `vectorized_elementwise_kernel<...CUDAFunctor_add<BF16>>` at ~21% of step time, grid = leaf.numel() per call).
- Signal: Seven independent attempts to eliminate this add (R70 foreach_add, R76 wgrad-output pool, R80 dgrad-output pool, R81 returning None for the b-grad slot, R95 leaf `Tensor.register_hook`, R99 C++ TORCH_LIBRARY autograd::Function, R100 `b.detach()` at the wrapper boundary) all rolled back: returning None violates the input-grad-presence contract; `register_hook` on the leaf fires AFTER `AccumulateGrad` in dispatch order and cannot pre-empt it; pool/arena swaps stabilize destination addresses but the engine still emits the per-leaf accumulate; detach + output-hook re-routes correctly only if the autograd Function's wgrad branch is gated off when `needs_input_grad[i]==False` (otherwise wgrad fires twice).
- Takeaway: PyTorch's `AccumulateGrad` add for a leaf input cannot be eliminated by destination-buffer pooling, leaf-side hooks, foreach batching, or returning None — the only contract-safe routes are (a) detach the leaf at the public-op boundary AND register the writeback hook on the FORWARD OUTPUT (not the leaf) AND gate the autograd Function's grad branch off for that slot to avoid double compute, or (b) collapse multiple per-call wgrad outputs into one tensor so the framework emits one add instead of N.
- Applicability: Apply when profiling shows a framework-emitted bf16/fp16 `AccumulateGrad` add as a top consumer for a leaf parameter that receives gradients from multiple per-step calls. Do NOT apply to leaves that already receive a single gradient tensor per step (the add then equals the unavoidable optimizer-side accumulate) or to inference-only paths.

### 2026-04-27 06:12 round-98 — ROLLED_BACK
- Context: gfx950 + ROCm/PyTorch + Triton kernels with per-step transient large-buffer allocations (dgrad/wgrad outputs, FP8 quant scratch) where graph-capture (manual `torch.cuda.CUDAGraph` R26/R98 or `torch.compile(mode='reduce-overhead')` R87) is considered to amortize ~1000+ HIP host dispatches/step.
- Signal: Three graph-capture attempts (R26 manual CUDAGraph, R87 TorchInductor reduce-overhead, R98 stable-pool-anchored CUDAGraph) all failed: R26 broke -50% on core shape because per-call PyTorch caching-allocator churn invalidated captured graph pointers; R87 correctness-failed because Inductor's online JIT could not stabilize FP8+grouped-shape signatures within the dynamic-shape path; R98 correctness-failed even with R56/R57/R77 weakref-stable input pointers as anchors because the per-call dgrad/wgrad output buffers still came from the caching allocator with non-stable addresses.
- Takeaway: Graph capture on ROCm requires ALL graph-input AND graph-output tensor addresses to be stable across replays — input stabilization via weakref-keyed quant caches is necessary but NOT sufficient; transient output buffers must also be backed by process-singleton pools, AND any in-graph autograd-emitted ops (e.g. `AccumulateGrad`) must be address-compatible with the captured DAG.
- Applicability: Apply when planning graph capture on ROCm: stabilize BOTH input (e.g. weakref quant cache) AND output (process-singleton pool) addresses, and validate that no autograd-engine-emitted op writes to a fresh per-call address inside the capture window. Do NOT attempt graph capture without first proving that every tensor crossing the capture boundary has a stable storage pointer across at least 3 consecutive replays.
